# VPS Guard v1 产品需求文档

## Problem Statement

VPS 管理者经常需要切换 SSH 端口、关闭不再需要的入口、按需放行端口并启用登录防护。手工编辑 OpenSSH、nftables 和 Fail2ban 配置容易遗漏 IPv6、覆盖第三方规则，或者因错误配置立即断开 SSH 且无法恢复。现有操作缺少统一的中文界面、应用前检查、可审计快照，以及在连接中断后仍能执行的自动回滚。

## Solution

提供一个面向 Debian 与 Ubuntu 的纯 Bash 安全管理工具。用户既可通过分层中文菜单完成常用操作，也可使用明确的 CLI 子命令和 dry-run。工具统一以 nftables 管理自身防火墙规则，提供 SSH 端口迁移与加固、密钥设置、端口规则、Fail2ban 预设、状态诊断、快照恢复和安装更新能力。

所有可能导致断网的变更都按事务执行：应用前备份并校验，设置独立于当前 SSH 会话的定时回滚，应用配置，要求用户从新会话验证，再确认提交并取消回滚。工具只管理自己创建的配置，不清空 Docker、VPN、控制面板或用户的第三方规则。

## User Stories

1. 作为 VPS 管理者，我希望通过中文编号菜单管理安全配置，从而不必记忆多套命令语法。
2. 作为熟悉自动化的管理员，我希望使用 CLI 子命令完成相同操作，从而可以重复执行和审计。
3. 作为谨慎的管理员，我希望使用 dry-run 查看命令、配置差异和影响范围，从而在写入前发现风险。
4. 作为 VPS 管理者，我希望工具检测发行版、版本、架构、权限和依赖，从而只在受支持环境中安全运行。
5. 作为 VPS 管理者，我希望缺失依赖时看到 APT 安装计划并单独确认，从而避免不知情的软件变更。
6. 作为 VPS 管理者，我希望修改 SSH 端口时暂时保留旧端口，从而能先验证新入口。
7. 作为 VPS 管理者，我希望 SSH 修改前进行语法检查和平滑重载，从而避免无效配置终止服务。
8. 作为 VPS 管理者，我希望高风险操作默认设置五分钟自动回滚，从而在断连后恢复可用配置。
9. 作为 VPS 管理者，我希望可选择三、五或十分钟回滚窗口，从而适应不同网络条件。
10. 作为 VPS 管理者，我希望从新 SSH 会话完成一次性确认后才关闭旧端口，从而证明新登录路径可用。
11. 作为 VPS 管理者，我希望查看当前生效的 SSH 配置和风险项，从而了解实际状态而非只看配置文件。
12. 作为 VPS 管理者，我希望选择性禁用密码登录、限制 root 登录和收紧认证参数，从而降低暴力破解风险。
13. 作为 VPS 管理者，我希望只有在密钥登录验证成功后才能禁用密码，从而避免锁死服务器。
14. 作为没有现成密钥的用户，我希望获得本机生成密钥并导入公钥的引导，从而安全启用密钥登录。
15. 作为特殊场景用户，我希望可在明确警告下于 VPS 生成加密私钥并安全取回，从而处理服务商未提供密钥入口的情况。
16. 作为安全管理员，我希望私钥和密码永不出现在终端输出或操作日志中，从而降低凭据泄露风险。
17. 作为 VPS 管理者，我希望启用入站默认拒绝、出站默认允许的防火墙基线，从而获得适合服务器的默认防护。
18. 作为网络管理员，我希望已建立连接、回环和必要 ICMP/ICMPv6 被正确处理，从而不破坏基本网络能力。
19. 作为 VPS 管理者，我希望防火墙自动保留已验证的 SSH 入口，从而避免启用时断连。
20. 作为 VPS 管理者，我希望用单端口、逗号列表、端口段或混合形式开放端口，从而快速表达业务需求。
21. 作为 VPS 管理者，我希望为规则选择入站或出站、TCP 或 UDP、IPv4、IPv6 或双栈，从而准确控制流量。
22. 作为 VPS 管理者，我希望端口规则可限制来源 IP、CIDR 或网卡，也可允许所有来源，从而适应公网和内网服务。
23. 作为 VPS 管理者，我希望重复开放已允许的端口时得到幂等提示，从而不会生成重复规则。
24. 作为 VPS 管理者，我希望关闭端口时删除工具创建的放行规则，从而在默认拒绝策略下自然关闭入口。
25. 作为 VPS 管理者，我希望关闭端口后仍看到本机监听进程警告，从而知道服务仍在运行。
26. 作为 VPS 管理者，我希望端口状态分别展示防火墙规则、本机监听和外部可达性是否已验证，从而不混淆三个概念。
27. 作为 VPS 管理者，我希望工具给出外部检测命令，从而验证云安全组、NAT 和公网路径。
28. 作为 VPS 管理者，我希望工具检测 UFW、iptables 和 firewalld，从而避免多个管理器混写 Netfilter 规则。
29. 作为 VPS 管理者，我希望检测到冲突防火墙时看到诊断并安全退出，从而不会破坏现有规则。
30. 作为 Docker、Podman 或 LXC 用户，我希望工具不修改容器的转发、NAT 和第三方规则，从而保持容器网络可用。
31. 作为 VPN 用户，我希望工具检测 WireGuard、OpenVPN 和 Tailscale 并提醒所需端口，从而避免误封隧道。
32. 作为控制面板用户，我希望工具提醒宝塔、1Panel 等服务的监听端口，从而能在启用防火墙前确认。
33. 作为 VPS 管理者，我希望使用宽松、标准、严格或自定义 Fail2ban 预设，从而按风险偏好配置 SSH 防护。
34. 作为 VPS 管理者，我希望标准预设默认采用有限重试和一小时封禁，从而兼顾防护与误封风险。
35. 作为 VPS 管理者，我希望可启用重复攻击者渐进封禁，从而提高持续攻击成本。
36. 作为 VPS 管理者，我希望当前管理 IP 可经确认加入白名单，从而降低封禁自己的风险。
37. 作为 VPS 管理者，我希望查看封禁列表并解封指定 IP，从而处理误封。
38. 作为 VPS 管理者，我希望所有配置修改前建立带时间戳和校验和的快照，从而可验证地恢复。
39. 作为 VPS 管理者，我希望默认保留最近十份快照并可调整，从而控制磁盘占用。
40. 作为 VPS 管理者，我希望查看快照差异并恢复指定版本，从而选择正确恢复点。
41. 作为 VPS 管理者，我希望“恢复全部”恢复最近完整快照，而不是盲目套用固定默认值，从而回到已知状态。
42. 作为 VPS 管理者，我希望另有明确的 SSH 重置为 22 选项，从而处理需要标准端口的恢复场景。
43. 作为 VPS 管理者，我希望停用防火墙只移除本工具管理的规则，从而不清空第三方配置。
44. 作为 VPS 管理者，我希望每个恢复或停用操作前看到针对性的风险警告，从而理解影响。
45. 作为审计者，我希望日志记录操作者、时间、变更摘要、结果和回滚状态，从而追踪管理行为。
46. 作为安全管理员，我希望备份仅 root 可读写且日志排除敏感数据，从而保护服务器信息。
47. 作为首次用户，我希望主菜单简洁并按 SSH、防火墙、Fail2ban、诊断、恢复等类别进入子菜单，从而快速找到功能。
48. 作为首次用户，我希望快速安全配置只提供少量推荐方案并可跳入相关子菜单，从而兼顾简洁与可定制性。
49. 作为 VPS 管理者，我希望安装、更新和卸载都先展示影响并要求确认，从而避免静默变更。
50. 作为 VPS 管理者，我希望更新前备份程序且不进行静默自动更新，从而避免未经审核的远程代码执行。
51. 作为开源用户，我希望 Release 包含版本化源码、单文件构建和 SHA-256 校验和，从而方便获取和验证。
52. 作为维护者，我希望错误输入、依赖失败或配置冲突返回明确退出码，从而便于自动化判断。
53. 作为维护者，我希望源码有解释风险与设计原因的中文注释，并有完整中文 Markdown 文档，从而便于审计和贡献。

## Implementation Decisions

- v1 正式支持 Debian 13、Debian 12 LTS、Ubuntu 22.04/24.04/26.04 LTS，以及 amd64、arm64。
- 核心使用 Bash 5.x，普通用户无需 Python、Node.js 或图形化菜单依赖。
- 项目采用模块化源码、单一入口；发布流程额外生成可分发的合并单文件版本。
- 用户界面为分层中文文本菜单；非交互终端自动关闭 ANSI 颜色。
- CLI 与菜单调用同一领域操作，支持 dry-run、稳定退出码和对高风险操作的显式确认参数。
- nftables 是唯一写入型防火墙后端。工具检测其他管理器但不与其混用；iptables 只用于现状识别和迁移提示。
- 工具拥有独立命名的规则范围，只修改自己创建的规则，不清空第三方表、容器链、转发或 NAT。
- 默认防火墙策略为入站拒绝、出站允许，并正确处理连接跟踪、回环及必要 ICMP/ICMPv6。
- 端口语法支持单值、列表、范围和混合输入；规则维度包括方向、协议、地址族、来源地址和可选接口。
- “关闭端口”表示撤销工具管理的放行规则；端口检查区分策略、监听和外部验证。
- SSH 端口迁移采用两阶段提交：保留旧端口、开放新端口、校验并重载、启动回滚、从新会话确认、关闭旧端口、取消回滚。
- SSH 加固是可选子菜单。禁用密码前必须验证目标用户可用密钥登录。
- 默认密钥流程只导入公钥。VPS 生成私钥是带警告的备用流程，私钥必须加密、取回后删除，且不得输出到日志。
- 高风险网络操作均支持三、五、十分钟的独立 systemd 定时回滚，默认五分钟；回滚不依赖当前 SSH 会话存活。
- 快照采用事务清单、权限与校验和，默认保留最近十份；恢复可选择指定快照。
- Fail2ban 默认只启用 sshd jail，使用 systemd 日志和 nftables 动作，提供宽松、标准、严格、渐进和自定义方案。
- 标准 Fail2ban 预设为十分钟观察窗口、五次失败、一小时封禁；不默认永久封禁。
- 环境预检检测容器、VPN、控制面板、监听端口、云代理和防火墙冲突，只提醒和保护，不主动管理第三方网络。
- 依赖仅可在用户确认后通过 Debian/Ubuntu 官方 APT 源安装；不添加第三方源、不执行远程脚本、不升级整个系统。
- 安装采用版本化目录和稳定命令入口；更新只检查和显式确认，不静默运行；卸载默认保留系统配置、快照和日志。
- 文档以中文为主，详细记录 CLI、dry-run、退出码、安全模型、恢复步骤、兼容性和已知限制；源码中文注释解释非显然决策。
- 项目公开发布于 GitHub，采用 MIT License，版权署名为 jLjT。

## Testing Decisions

- 测试只断言用户可见行为，不锁定内部 Bash 函数、命令排列或模块组织。
- 主要测试缝隙是完整调用用户可见的命令入口，在临时文件系统和受控系统命令替身上运行场景。
- 端到端 CLI 场景验证交互输入、中文输出、退出码、dry-run 计划、生成配置和事务状态。
- SSH 场景覆盖端口验证、配置语法失败、旧端口保留、新会话确认、超时回滚和成功提交。
- 防火墙场景覆盖端口解析、IPv4/IPv6、TCP/UDP、方向、来源、接口、幂等与冲突管理器。
- Fail2ban 场景覆盖各预设、自定义边界、白名单、状态和解封。
- 快照场景覆盖权限、清单、校验和、保留策略、指定恢复和部分失败后的原子性。
- 环境诊断场景覆盖 Docker/VPN/面板识别，并断言第三方规则不会被修改。
- 静态门禁使用 ShellCheck 和统一格式检查；行为测试使用 Bats 或等价 Bash 测试框架。
- 发布前在受支持系统矩阵的隔离虚拟机中进行少量真实验收，覆盖 systemd、sshd、nftables、Fail2ban 和真实断连回滚。
- 容器测试不能替代需要内核 Netfilter、完整 systemd 与真实 SSH 会话的虚拟机验收。

## Out of Scope

- v1 不支持 RHEL、Rocky Linux、AlmaLinux、CentOS、Arch Linux 或非 Linux 系统。
- v1 不写入 UFW、iptables 或 firewalld 规则，也不自动迁移复杂的第三方规则集。
- v1 不管理云厂商安全组、外部 NAT、路由器端口转发或服务商控制台。
- v1 不管理 Docker/Podman/LXC 的转发、NAT 和容器间网络策略。
- v1 不承担通用路由器、防火墙网关或 NAT 管理功能。
- v1 不保证仅凭服务器本机即可判断公网端口可达。
- v1 不提供图形界面、Web 面板或桌面应用。
- v1 不进行静默自动更新，也不从第三方软件源安装依赖。
- v1 不默认启用永久 Fail2ban 封禁或激进的出站阻断策略。
- v1 不删除、打印或上传用户已有私钥。

## Further Notes

- 所有风险提示必须说明最坏后果，并建议用户预先确认云控制台、串行控制台或救援模式可用。
- SSH 监听成功不等于外部可达；确认流程必须来自独立的新 SSH 会话。
- IPv6 不应通过粗暴禁用规避规则设计；双栈主机必须明确处理 IPv4 与 IPv6。
- 快速安全配置保持简洁，高级参数进入对应子菜单，子菜单完成后可返回原流程继续。
- 首个正式 Release 必须在至少一个真实 Debian 与一个真实 Ubuntu 隔离环境完成断连回滚验收。
