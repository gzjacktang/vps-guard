# 安全模型

## 所有权边界

VPS Guard 只备份和恢复登记为“受管路径”的配置。快照中的绝对路径必须通过路径穿越检查；快照时不存在的受管文件会记录为 `missing`，因此回滚能够删除操作后新建的该文件。工具拒绝按 `missing` 记录递归删除目录。

## 快照完整性

- 数据目录默认为 `/var/lib/vps-guard`，目录权限为 `0700`。
- 每个普通文件记录 SHA-256 与原权限。
- 恢复前验证全部文件，任一校验失败均不开始覆盖。
- 文件先在目标目录准备，再逐个替换；提交中途失败时恢复已经替换的旧文件。
- 默认只保留最近 10 份快照，可设置为 1–100。

## 自动回滚

回滚由 `systemd-run` 创建临时单元，不依赖发起操作的 SSH 会话存活。状态包括等待确认、已确认、执行中、已回滚、失败和调度失败。确认操作先停止 systemd 单元，再写入已确认状态，避免“显示成功但任务仍运行”。

## 快速组合事务

快速向导不是依次调用三个各自带快照的高层命令，而是组合事务的唯一所有者。所有 SSH、nftables 与 Fail2ban 候选在写入前完成语法、所有权和冲突校验；最终确认后才创建一份覆盖全部受管路径的快照和一个自动回滚任务。任何持久写入均发生在回滚调度成功之后。

标准方案使用 SSH 旧、新端口过渡集合同时渲染防火墙和 Fail2ban。磁盘或运行时任一步失败都会恢复起始快照，并分别重载自有 nftables 表、sshd 和 Fail2ban；恢复到原先没有自有 jail 时也重启 Fail2ban 以清除内存状态。只有完整恢复且成功取消计时器时，工具才会宣称立即恢复完成，否则保留计时任务并要求从带外控制台处理。

向导令牌保存方案、快照、回滚、旧端口和目标端口，状态目录为 root-only。改变 SSH 端口时，提交必须来自目标端口的真实 `SSH_CONNECTION`；普通回滚确认拒绝 `wizard-*` hook，避免旧会话跳过证明。超时任务与确认共享全局配置事务锁，不能并发覆盖同一组受管文件。完整用户流程见 [快速安全配置](QUICK-START.md)。

## 审计与敏感数据

审计日志默认为 `/var/log/vps-guard/audit.log`，目录 `0700`、文件 `0600`。记录 root UID，并在通过 sudo 执行时记录经过格式校验的 `SUDO_USER`。日志不得记录密码、私钥、用户自由输入的标签或认证日志正文；调用模块只能提交动作标识、结果、快照/回滚标识和预定义原因。

发布构建额外扫描源码包和合并单文件，拒绝私钥标记、常见令牌、日志、认证文件、归档绝对路径、路径穿越、链接和特殊文件。仓库中的网络示例只使用文档保留地址，不提交真实服务器地址、IP、备份、状态目录或原始日志。

## 安装与更新供应链

稳定命令是固定 launcher，它执行 `current` 指向的版本目录；直接把模块入口软链到 `/usr/local/sbin` 会使 `BASH_SOURCE` 从错误目录寻找 `lib`，因此明确禁止。systemd 回滚使用不可变版本目录中的真实脚本，更新保留旧版本；活动或失败待重试的回滚会阻断卸载。

`update check` 只下载官方 GitHub Release 元数据，绝不下载 payload 或执行脚本。安装/更新只接受管理员已经下载、校验并审阅的本地源码目录，替换前复制现有程序备份，暂存版本必须通过 Bash 语法和版本健康检查才切换入口。SHA-256 不是签名，不能证明发布账号未被攻破。完整操作见 [安装、手动更新与卸载](INSTALLATION.md)。

卸载默认不触碰 `/etc` 受管配置、live nftables/sshd/Fail2ban、快照或日志。数据清理也只删除 `/var/lib/vps-guard` 与审计日志，且要求第二次固定短语；要停用安全配置必须先走各模块受保护的 disable/restore 流程。

## 第三方网络保护

任何防火墙写入前都必须通过只读环境预检。UFW、firewalld、非容器 iptables 规则或无法读取的管理状态会返回退出码 3 并默认阻断。已有 nftables 表、容器链、VPN 和控制面板只被检测和提醒，VPS Guard 不接管其生命周期。

VPS Guard 不修改 `FORWARD`、NAT、Docker/Podman/LXC/CNI/Kubernetes 链、VPN 配置或第三方 nftables 表。检测规则、命令清单、误判处理和自动化退出码详见 [网络环境预检说明](PREFLIGHT.md)。

## 防火墙事务

防火墙只拥有 `table inet vps_guard`、`/etc/nftables.d/vps-guard.nft`、`/etc/vps-guard/firewall.conf` 和 `/etc/nftables.conf` 中一条精确标记的 include。候选规则必须先通过 `nft -c`；确认后才创建快照并应用。应用失败、回滚调度失败或超时都会恢复磁盘配置并同步恢复内核中的自有表。

高级规则以严格校验的原子记录保存在 root-only 状态文件中，不使用 `eval`。协议与双栈组合在写入前展开；接口、地址和端口均拒绝控制字符与 nftables 文本注入。出站 drop 前保留已建立/相关连接，降低当前管理会话被瞬间切断的风险，但新连接仍可能立即失败，因此回滚调度必须先于 live reload。

`/etc/vps-guard/firewall.conf` 是防火墙所有权记录。没有有效状态文件时，即使发现的对象名称恰好是 `inet vps_guard`，也一律视为第三方或归属不明并返回冲突码 3，禁止接管。

启用时同时保留当前 `SSH_CONNECTION` 服务端端口和 `sshd -T` 配置端口。存在未确认的防火墙事务时禁止叠加第二个事务。详细流程见 [nftables 防火墙说明](FIREWALL.md)。

## SSH 两阶段事务

SSH 迁移把 `SSH_CONNECTION` 当作已验证会话证据，把 `sshd -T` 当作磁盘配置的生效结果；两者用途不同，不能互相替代。迁移阶段先启动 systemd 回滚，再同时写入旧、新端口并同步防火墙。`sshd -t`、生效端口、sshd 监听、服务 reload 或防火墙任一检查失败都会立即恢复快照。

只有目标新端口会话才能通过 `ssh confirm` 关闭旧端口。关联的 `ssh-firewall` 回滚令牌拒绝普通 `rollback confirm`，避免在旧会话或本机监听检查后绕过证明步骤。SSH 迁移或选择性恢复等待确认时，所有普通防火墙写入也被阻断，避免不同快照的计时器交叉覆盖。超时恢复文件后还会同步 reload sshd 与自有 nftables 运行时。详细状态、受管文件和恢复入口见 [SSH 端口两阶段迁移](SSH.md)。

## SSH 密钥与认证证明

公钥不是私钥，但工具仍不在状态摘要或审计日志记录公钥正文，只记录 SHA-256 指纹。私钥与口令绝不写入审计：客户端私钥不上传；服务器备用流程由 `ssh-keygen` 直接从 TTY 读取口令，不放入参数，并在确认或 10 分钟超时后删除。

禁用密码不以“文件中存在 authorized_keys”作为成功依据。工具通过 OpenSSH `ExposeAuthInfo` 提供的 `SSH_USER_AUTH`，校验当前新会话确为目标用户、使用 publickey 且指纹匹配。加固提交与定时回滚共享同一锁；普通回滚确认无法绕过。详细边界见 [SSH 密钥设置与可选加固](SSH-HARDENING.md)。

公钥导入对认证 drop-in 的临时修改也采用确认/自动撤销：客户端导入默认 5 分钟，服务器备用流程 10 分钟。用户目录内的密钥写入降权为目标用户执行；异常属主、链接、`AuthorizedKeysCommand` 和可能依赖密码的组合 `AuthenticationMethods` 默认拒绝。防火墙、SSH 端口、密钥与加固的准备阶段共用全局配置事务锁，避免两个快照或计时器交叉覆盖。定时回滚同样获取该锁；等待超过 60 秒会明确失败并由 systemd 每 30 秒重试，而不是永久死等或并发覆盖。

该证明用于降低管理员误锁风险，并不构成针对已拥有 `root` 或完整 `sudo` 权限之恶意用户的隔离边界；管理员本就可以改写 sshd 配置、进程环境和工具状态。VPS Guard 的威胁模型要求 root 状态目录、受管配置与审计文件只由可信管理员控制。

## Fail2ban 所有权与联动

VPS Guard 只拥有 `/etc/fail2ban/jail.d/vps-guard.local`，且以所有权标记、root 属主、`0600` 文件权限和安全父目录共同判定归属；状态模糊时拒绝接管。工具不卸载 Fail2ban、不删除其他 jail。候选配置先在临时配置树运行 `fail2ban-client -t`，再在目标目录内暂存并原子 rename；live 写入后再次校验、重启并检查 sshd jail，失败则恢复快照。SSH 端口迁移的过渡与提交阶段会同步该 jail 的端口，使旧、新端口的可达窗口与防护窗口一致。

默认使用 `backend = systemd` 和 `nftables[type=multiport]`。这会创建 Fail2ban 自己的 nftables 对象，不取得 `table inet vps_guard` 所有权，也不清空第三方规则。当前管理 IP 只在检测成功并得到明确同意后加入白名单；工具不会从任意网络推断可信网段。具体预设、限制和真实系统验证门禁见 [Fail2ban SSH 防护](FAIL2BAN.md)。
